- **Bug Fix: tailwindPlugin export + CLI polish (2026-03-14):** Root cause of `tailwindPlugin` SyntaxError: published npm v1.8.2 predated the theme work, so downstream projects got a `vite.js` without `tailwindPlugin`. Fixes: (1) Bumped engine version `1.8.2` → `1.9.0` to reflect new tailwind/theme exports. (2) Added `"./vite.js": "./vite.js"` to exports map so both `@deckio/deck-engine/vite` and `@deckio/deck-engine/vite.js` resolve correctly. (3) Updated scaffolder dep `^1.8.2` → `^1.9.0`. CLI UX overhaul: ANSI-colored prompts (`fmt` helpers), default accent changed `#3fb950` → `#6366f1` (indigo-500, more modern), accent color swatch preview via true-color ANSI, theme picker shows descriptions (deep space / projection / editorial neutral), styled success output. No external deps added (no gum). 94 tests passing.
- **Critical Bug Fix: Production build "dirname" error (2026-03-14):** `index.js` re-exported `getAvailableThemes`, `resolveTheme`, `DEFAULT_THEME`, `BUILTIN_THEMES` from `themes/theme-loader.js`, which imports Node.js builtins (`fs`, `path`, `url`). Rollup can't resolve these browser-side during production builds — dev server hid the issue because Vite doesn't bundle in dev mode. Fix: (1) Removed theme-loader re-exports from `index.js` — browser entry now exports only React components, hooks, and constants. (2) Added re-exports of all theme-loader utilities (`resolveTheme`, `getAvailableThemes`, `DEFAULT_THEME`, `BUILTIN_THEMES`) from `vite.js` so Node.js consumers access them via `@deckio/deck-engine/vite`. Theme-loader also remains directly importable via `@deckio/deck-engine/themes/theme-loader`. **Key lesson:** Never re-export Node.js-only modules from the main browser entry point. The browser/Node boundary must be enforced at the exports map level. 94 tests passing, production build verified with scaffolded project.
- **Critical Bug Fix: Scaffolder version drift / ETARGET (2026-03-14):** `utils.mjs` hardcoded `'@deckio/deck-engine': '^1.9.0'` but v1.9.0 was never published (latest npm = 1.8.2). Every scaffolded project failed `npm install`. Root cause: version bump in engine package.json without npm publish, and scaffolder used a hardcoded string instead of reading dynamically. Fix: (1) Reverted engine version `1.9.0` → `1.8.2`. (2) Added `resolveEngineRef(projectDir)` to `utils.mjs` — reads the engine's actual version from its `package.json` at scaffold time. (3) In local dev (monorepo), generates `file:` protocol references so npm install resolves from the local filesystem, bypassing the registry entirely. (4) `packageJson()` now accepts optional `engineRef` parameter; defaults to `resolveEngineRef()`. (5) Cleaned up stale test project. 7 new tests for resolveEngineRef. 101 tests passing. E2E verified: scaffold → npm install → vite dev → vite build all pass. **Key lesson:** Never hardcode dependency versions in scaffolder templates. Always read from the source package.json. For local dev, use `file:` protocol to decouple from npm publish status. Second scaffolder bug that shipped — must add E2E verification to publish workflow.
- **shadcn/ui Design System Integration (2026-03-14):** Added opt-in `designSystem` field to scaffolder (`"none" | "shadcn"`, default `"none"`). Separate from `theme` — you can use theme `"shadcn"` without the component system, or (default) get both. CLI asks "Include shadcn/ui components?" when theme is `"shadcn"`, defaults Yes. When enabled: (1) `packageJson()` adds `class-variance-authority`, `clsx`, `tailwind-merge` as deps. (2) Generates `components.json` with `rsc: false`, `tsx: false` (JSX project, no RSC). (3) Creates `src/lib/utils.js` with `cn()` utility (clsx + tailwind-merge). (4) Creates `jsconfig.json` with `@/*` → `./src/*` path alias (required by shadcn CLI). (5) `viteConfig()` adds `resolve.alias` mapping `@` → `src/`. (6) `deckConfig()` includes `designSystem: 'shadcn'` field. (7) README gets shadcn usage section. (8) Creates empty `src/components/ui/` directory. Non-interactive mode reads `DECK_DESIGN_SYSTEM` env var, defaults to `"shadcn"` when theme is `"shadcn"`. Non-shadcn themes get zero shadcn artifacts. `VITE_CONFIG` constant replaced with `viteConfig()` function for conditional alias generation. **Key lesson:** shadcn CLI requires `jsconfig.json` (or `tsconfig.json`) for path resolution — without it, `npx shadcn add` fails with "Couldn't find tsconfig.json". Pre-generating files instead of running `npx shadcn init` is more reliable and faster. 130 tests passing (was 101). Full E2E verified: scaffold → npm install → dev server → production build → `npx shadcn add button` → button renders in slide → build succeeds with button import.
